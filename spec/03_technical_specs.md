# REFLEX LABORATORY ゲーム仕様書
## 03. 高精度計測技術とモニター補正仕様

反応速度ベンチマークとして、最も致命的となる「ブラウザ、OS、および表示デバイス(モニター)起因の誤差」を徹底的に取り除くためのシステム実装レベルの仕様を定義します。

---

### 1. マイクロ秒単位の高精度時間計測
Javascript標準の `Date.now()` (ミリ秒精度、OS依存の揺らぎあり)に起因するミリ秒未満の丸め誤差や計測の揺らぎを回避するため、以下の手法を用います。

- **`performance.now()` の完全採用**
  - 高精度タイムスタンプ（High Resolution Time）を取得する目的で `performance.now()` を計測イベント（開始、終了）の基準タイムに全面的に採用します。
  - ナノ秒（μs）オーダーまで検出可能な高精細な内部タイマーから直接差分を引き算し、丸め歪みのない計測を行います。

---

### 2. インプットラグの極限低減：`PointerDown` の採用
通常のPC用ブラウザゲームで多用される「`Click`（マウスを押し下げて離すまでの計測）」や「`MouseUp`」では、クリック操作完了までに数十ミリ秒の物理的な遅延が発生します。

- **仕様**:
  - デバイスに接触、あるいはボタンが押下された最速の瞬間をミリ秒単位で検知・優先するため、イベントハンドラには `PointerDown`（タッチスクリーン、ペン、マウスの統合イベント）を採用します。
  - タッチ端末（iPadなどのスマートフォン・タブレット）では `touchstart` と同じく遅延を最小に抑えられます。
  - テストエリアの `PointerDown` イベント発生のコールバック内で最速で現在時刻（終了時間）を引き出し、差分を確定します。

---

### 3. デバイス別補正アルゴリズム（システム・キャリブレーション）
表示モニターのリフレッシュレート（Hz、秒間の書き換え枚数）が異なるだけで、画面上に円が出現してから私たちの網膜に像が届くまでの物理的な限界時間が変化します。
例えば、60Hz（表示間隔 約16.6ms）と 240Hz（表示間隔 約4.1ms）では、表示更新だけで物理的なディレイに最大 「約12.5ms」 のギャップが必然的に生じます。
このハンディキャップ（不公平）を解消するため、本作は**240Hzモニター**を絶対基準 (ゼロ・オフセット) と定め、各リフレッシュレートに応じた補正オフセット値（ms）を自動適用します。

#### ① オフセット補正数式
選択（または実測）されたリフレッシュレート（Hz）に対して適用される補正時間（ミリ秒）:

$$\text{補正オフセット(ms)} = \left( \frac{500}{\text{選択したリフレッシュレート (Hz)}} \right) - 2.1$$

*※240Hzの時にちょうど $0$ ms近辺となるように調整された理論値式です。*

- **具体例 (計算値)**:
  - **60Hz**: $\approx 6.2 \text{ ms}$ を基準評価値に加算（遅延ハンデを加味し、評価される目標値が緩やかになります）
  - **144Hz**: $\approx 1.4 \text{ ms}$ 加算
  - **240Hz**: $0 \text{ ms}$ (絶対基準)
  - **360Hz**: $\approx -0.7 \text{ ms}$ （基準を厳しく減算）
  - **500Hz**: $\approx -1.1 \text{ ms}$

#### ② 評価基準一覧（240Hz基準の場合）
このオフセット補正時間を各指標の境界線に足すことで、モニター種別を問わず、同じ身体能力であれば同一の「評価ランク（ランクバッジ）」に属するように設計します。

| 評価ランク | 評価閾値 (240Hz基準) | 特徴・期待スコア |
| :--- | :--- | :--- |
| **最高** (Godlike) | $150 \text{ ms}$ 以下 | プロゲーマー・人間離れした極限のタイミング |
| **とても良い** | $150 \text{ ms}$ 超 〜 $170 \text{ ms}$ | 非常に早い、高い集中力と優れた機材環境 |
| **良い** | $170 \text{ ms}$ 超 〜 $190 \text{ ms}$ | 一般的なゲーマーレベル、機敏なレスポンス |
| **平均以上** | $190 \text{ ms}$ 超 〜 $220 \text{ ms}$ | 優秀。平均的な状態より明らかに速い |
| **平均** | $220 \text{ ms}$ 超 〜 $260 \text{ ms}$ | 一般的な健康状態かつ標準的な反応速度 |
| **平均以下** | $260 \text{ ms}$ 超 〜 $320 \text{ ms}$ | やや反応がのんびりしている、または遅延機材環境 |
| **遅い** | $320 \text{ ms}$ 超 〜 $400 \text{ ms}$ | 指先の硬さ、または画面の処理遅延などがある |
| **とても遅い** | $400 \text{ ms}$ 超 〜 $500 \text{ ms}$ | 著しく反応が鈍いか、不快なデバイス遅延が発生中 |
| **練習が必要** | $500 \text{ ms}$ 超 | ヤマ勘を排した実測値。要トレーニング |

### 4. 描画同期 requestAnimationFrame の接続
JavaScript での非同期な setTimeout 等による標的出力を排し、モニターのレンダリングサイクル（描画完了の直前タイミング）にジャストフィットさせるために `requestAnimationFrame` で描画関数と基準時間測定開始のイベントリスナー、レンダラースレッドを同期します。
これによって、「コードが実行された時間」と「画面に実際に光が到達した時間」のずれ（Jitter：ジッター）を極力抑えます。
